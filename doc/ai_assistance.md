### 第 2 轮 — 游戏类型选择：为什么是塔防

**我:** 做游戏的话，什么样的游戏最适合展示面向对象设计？

**AI:** 从三个维度评判每种候选游戏：

**维度一：继承深度。** 贪吃蛇最多建一个 SnakeSegment 基类，蛇头蛇身行为差异极小——"为了继承而继承"。俄罗斯方块有 7 种方块，但差异仅是形状数据不同，旋转和碰撞逻辑完全一样——"用继承替代了配置数据"是反模式。塔防有三个在**行为层面**有本质差异的实体类别：敌人（导航+生存）、防御塔（感知+攻击）、子弹（飞行+碰撞）。这三个行为域互不重叠，所以需要三条独立的中间抽象基类。

**维度二：行为多态。** 塔防同一行 `for enemy in enemies: enemy.update(dt)` 产生三种完全不同
的移动体验：哥布林快速穿梭、兽人稳步推进、Boss 缓慢但血量纹丝不动。贪吃蛇的多态几乎为零。

**维度三：设计模式空间。** 塔防天然映射四种模式：策略模式（不同塔=不同攻击策略）、模板方法
（GameObject.update 检查存活后调用子类钩子）、工厂模式（波次系统按配置生产敌人）、组合优
于继承（Game 拥有 GameMap）。四种模式没有一个是"硬塞"的。

**结论：** 塔防在最少冗余代码的前提下展示最多 OOP 概念。RPG 虽然也能做到，但容易因系统膨胀
（装备/技能/对话）偏离重点——老师评分看的是 OOP 设计质量，不是游戏体量。

**我:** 塔防听起来很合适。继续。

### 第 3 轮 — 技术栈选择：用已学的库就够了

**我:** 课程学了 NumPy、Matplotlib、Pandas。做游戏需要装 pygame 吗？

**AI:** 不需要。Matplotlib 有三个常被忽视的游戏能力：

1. **FuncAnimation**（Day5 学过）：`FuncAnimation(fig, update, interval=80)` 以固定间隔反复调用回调函数——这就是游戏主循环。

2. **事件系统**：`mpl_connect('key_press_event', handler)` 捕获键盘，`mpl_connect('button_press_event', handler)` 捕获鼠标。覆盖塔防全部输入需求。

3. **绘图 API**：`ax.scatter()` 画角色，`ax.add_patch(Rectangle(...))` 画地图格和血条，`plt.Circle()` 画射程圈。塔防的所有视觉元素都可以用这些基本图元实现。

**为什么不装 pygame？** 首先课程环境没有安装——提交的作业如果老师需要先 `pip install pygame` 才能运行，操作层面就不合格。更深层的原因：pygame 封装了太多东西。`pygame.sprite.Sprite` 帮你做好了碰撞检测，`pygame.time.Clock` 帮你做好了帧率控制。用 pygame 的话，大量代码是"调用 pygame API"而非"自己设计类和方法"。老师要看的是**你如何设计 GameObject 的 update 流程、如何管理实体列表、如何组织每帧的逻辑顺序**——这些在 pygame 中都被遮住了。

用 matplotlib 从零搭建，意味着你必须自己决定：帧率取多少（12.5 FPS，对塔防足够）、更新顺序是什么（生成→移动→瞄准→开火→碰撞→判定）、如何管理实体生命周期（死亡清理）、如何在无精灵系统下实现 Z-order（按 path_progress 排序再绘制）。这些决策本身就是**架构设计的体现**——评分的核心。

**局限承认：** matplotlib 的帧率天花板 ~12.5 FPS（50+ 个 artist 时）、无增量渲染（每帧全清全绘）、无原生碰撞检测（自己写欧氏距离）、输入精度有限。但对塔防游戏来说，这些都不是问题。

### 第 4 轮 — 架构设计：三层继承 + 四种设计模式

**我:** 帮我设计塔防的类层次结构。

**AI:** 提出三层继承链架构：

```
GameObject (ABC)           ← 模板方法模式：update() 定义流程框架
├── Enemy (ABC)           ← 策略模式：子类定义属性差异
│   ├── Goblin  (快速/低血)
│   ├── Orc     (中速/中血)
│   └── Boss    (慢速/高血)
├── Tower (ABC)           ← 策略模式：子类定义攻击策略
│   ├── ArrowTower   (快速/单体)
│   ├── CannonTower  (慢速/溅射)
│   └── IceTower     (中速/减速)
└── Projectile (ABC)      ← 策略模式：子类定义命中效果
    ├── Arrow       (直伤)
    ├── Cannonball  (溅射)
    └── IceShard    (减速)
```

辅助类（不继承 GameObject）：GameMap（网格地图+路径）、Wave/WaveFactory（工厂管理波次）、
TowerRegistry（注册表）、GameStats（Pandas 数据分析）、Game（编排层）。

**我:** 为什么是三层而不是两层或四层？为什么 Map 不继承 GameObject？

**AI:** 如果只有两层（GameObject 直接到具体子类），Enemy/Tower/Projectile 的所有方法都得塞进
GameObject。Goblin 会继承 `_find_target()`（但敌人不需要选目标），ArrowTower 会继承
`_move_along_path()`（但塔从来不会移动）——违反**接口隔离原则**，是"胖基类"反模式。

如果做四层（在 Enemy 和 Goblin 之间再加一层），需要回答"中间这一层代表什么有意义的抽象"。
如果三种敌人都没有需要独占中间层的新行为，那这一层就是空壳。"跑得快"和"慢慢走"在代码层
面是同一个 `_do_update()` 方法——只是 speed 参数不同。"用继承表达数据差异"是反模式。

**三层是最优**：第 1 层（GameObject）贡献**跨领域共性**（存活管理+坐标+模板方法）；第 2 层
（Enemy/Tower/Projectile）贡献**领域专属行为**（移动 vs 瞄准 vs 飞行）；第 3 层（具体子类）
贡献**数据差异化**（不同的 HP/速度/射程/冷却）。三层各司其职。

**Map 不继承 GameObject**是因为它不参与游戏循环。GameObject 的特征是 `for obj in objects: obj.update(dt)` 可遍历更新。Map 是静态配置（网格/路径/可建造区域）——它在游戏开始加载一次后不再改变。如果强制继承，要么空方法（违反契约），要么抛异常（说明继承就错了）。正确做法是**组合**：Game 拥有 GameMap——"游戏有一张地图"是 has-a，不是 is-a。

**我:** 11 个类 + 3 条链 + 明确的设计考量。架构扎实。继续细化。

### 第 5 轮 — 设计模式详解

**我:** 四种设计模式在代码中怎么体现的？和 Day6 课程内容的对应关系是什么？

**AI:** 逐一说明：

**策略模式（Tower 继承链）：**
```python
class Tower(GameObject, ABC):
    @abstractmethod
    def _create_projectile(self, target) -> Projectile: ...

class ArrowTower(Tower):
    def _create_projectile(self, target):
        return Arrow(self.x, self.y, target)  # 直伤策略

class CannonTower(Tower):
    def _create_projectile(self, target):
        return Cannonball(self.x, self.y, target, self.splash_radius)  # 溅射策略
```
基类定义策略接口 `_create_projectile()`，子类返回不同 Projectile。Game 主循环通过
`for tower in towers: tower.update()` 多态调用，不关心具体类型。

**与 Day6 的对应：** Grader.grade(score) ↔ Tower._create_projectile(target)；
PassFailGrader（最简单策略）↔ ArrowTower；LetterGrader（复杂判定）↔ CannonTower；
for grader in graders: grader.grade(score) ↔ for tower in towers: tower.update()。
Day6 的策略差异体现为**返回值不同**（字符串），塔防体现为**副作用不同**（创建不同子弹）。
两种都是策略模式：前者是计算型策略，后者是行为型策略。

**模板方法模式（GameObject.update）：**
```python
class GameObject(ABC):
    def update(self, dt, **kwargs):
        if not self._alive:          # 固定步骤
            return
        self._do_update(dt, **kwargs) # 可变步骤

    @abstractmethod
    def _do_update(self, dt, **kwargs): ...
```
三个子类的 `_do_update()` 实现了完全不同但被同一行代码触发（`obj.update(dt)`）的行为。
"死亡后不更新"的逻辑只写一次，所有 9 个子类自动受益。如果未来加入"眩晕"状态，只改模板
方法一次就行。

**工厂模式（WaveFactory + TowerRegistry）：**
```python
class WaveFactory:
    @classmethod
    def create_wave(cls, wave_number):
        return Wave(wave_number, cls._configs[wave_number - 1])
```
核心价值：**配置与逻辑分离**。WAVE_CONFIGS 是纯数据，修改第 5 波 BOSS 数量只需改一个数字；
Wave 是纯逻辑，不关心自己属于第几波。Day6 的 `__init__` 校验是工厂模式的基础——塔防项目
把这个概念提升了一级抽象，封装进独立的工厂类。

**组合优于继承（Game 拥有 GameMap）：**
Game 不继承 GameMap，而是拥有 GameMap 实例 `self._map = GameMap()`。这允许通过构造函数
注入不同地图配置，不改 Game 类代码。Day6 的 CourseTaskSystem 组合 tasks 和 students 列表
就是同一思想——塔防项目是它的"游戏版本放大"。

**我:** 四种模式都正确应用，和 Day6 的对应清晰。这个可以在说明文档里重点展示。

### 第 7 轮 — 敌人参数：数学推导

**我:** 三种敌人的参数怎么定？要保证能打死，又不能太容易。

**AI:** 以箭塔为基准反推所有数值。核心模型：一座箭塔（射程 3.0，冷却 0.5s，伤害 18）放在
路径旁 1 格处。

**几何模型：** 塔到路径垂直距离 d=1，射程 R=3.0，路径覆盖长度 = 2√(R²-d²) = 2√8 ≈ 5.657 格。

**时间与伤害模型：**

| 敌人 | HP | 速度 | 停留时间 | 射击次数 | 总伤害 | 差额 | 所需塔数 |
|------|-----|------|----------|----------|--------|------|----------|
| 哥布林 | 60 | 2.5 | 2.26s | 5发 | 90 | +30(+50%) | 1座 |
| 兽人 | 180 | 1.5 | 3.77s | 8发 | 144 | -36(-20%) | 2座(288伤) |
| Boss | 600 | 0.9 | 6.29s | 13发 | 234 | -366(-61%) | 3座(702伤) |

**推导逻辑：** 哥布林 HP=60 保证 1 座箭塔在射程内恰好可击杀（5 发 90 伤，溢出 50% 作为容错）。
兽人 HP=180 使 1 座塔 8 发 144 伤不够，**强制玩家布置多塔协同**——这是关键设计意图。
Boss HP=600 需要 3+ 座塔，但冰塔减速后有效伤害翻 2.5 倍，实际 3-4 座塔加 1 冰塔即可。

**我:** Boss 600 血会不会太高？

**AI:** 冰塔减速 40% 后 Boss 速度降至 0.36，在箭塔射程内停留从 6.3s 延到 15.7s。2 座箭塔
在减速下可各射 31 发 = 62×18 = 1116 伤 >> 600。冰塔只花 75g，但让其他塔的 Boss 输出翻了
2.5 倍——这正是引导玩家理解减速价值的设计。

### 第 8 轮 — 塔参数：三角定位 + DPS 设计

**我:** 塔的参数按什么逻辑设定？

**AI:** 三种塔形成"快攻-重击-控制"三角。关键设计决策：**箭塔 DPS=36，炮塔 DPS≈37，
单体 DPS 几乎相同。**这不是巧合——目的是"单目标等价，差异化在其他维度"。

**实际效果差异：**
- **攻击速度差异 → 伤害粒度不同：** 箭塔"细水长流"（每 0.5s/18 伤），炮塔"重拳出击"（每 1.5s/55 伤）。炮塔打残血哥布林浪费 91% 伤害（过杀），箭塔仅浪费 72%。
- **射程差异 → 有效窗口不同：** 炮塔射程 2.5 的路径覆盖比箭塔 3.0 少 19%，放置要求更苛刻。
- **溅射 → 对群爆发：** 炮塔溅射半径 1.0，当 N 个敌人集群通过时：有效 DPS = 36.7 + (N-1)×18.3。N=3 时 73 DPS（超箭塔 104%）。

**冰塔 DPS 只有 12 但不需要加强：**
减速 40% + 100% 覆盖率（冷却 1.0s < 持续 2.0s），伤害放大倍数 = 1/0.4 = 2.5×。
数值实例——冰塔 + 2 箭塔 vs Boss：
- 无冰塔：2 箭塔各 13 发 → 468 伤（不够）
- 有冰塔：2 箭塔各 31 发 → 1116 伤（溢出）
冰塔直接贡献 75 伤，间接贡献 573 伤——**是其自身伤害的 7.6 倍**。如果冰塔 DPS 也高，
它就成了唯一最优解。低 DPS 确保了它必须是**辅助定位**。

**我:** 三种塔各有定位，数值逻辑清晰。

### 第 10 轮 — 游戏主循环

**我:** matplotlib 的 FuncAnimation 怎么做游戏循环？

**AI:** 核心实现：
```python
FuncAnimation(self._fig, self._update, interval=80,
              cache_frame_data=False, save_count=0)
```
每帧 ~80ms 执行 5 个子步骤（顺序经设计）：
1. 波次敌人生成（_update_spawning）
2. 敌人移动 + 基地检查（_update_enemies）
3. 塔冷却 + 瞄准 + 开火（_update_towers）
4. 子弹飞行 + 碰撞 + 击杀记录（_update_projectiles）
5. 胜负判定（_check_wave_complete / _check_defeat）

**顺序不能乱：** 先生成→再移动→塔才能在正确位置选目标→子弹才能在移动后判断碰撞。
调换顺序会导致"塔瞄准上一帧位置"的 bug。12.5 FPS 对塔防足够（敌人每帧移 0.2 格，平滑）。

**我:** 继续写代码吧。

### 第 17 轮 — Bug 修复：第 2 波只出 4 只怪（2026-07-23）

**我:** 第 2 波出了几只怪就不出了！

**AI:** 定位到 wave.py 第 109 行的 `_counters` 字典。该字典用敌人类型字符串 `'goblin'`
做 key。第 2 波配置是两批哥布林：
```python
[(0.0, 'goblin', 4), (2.0, 'goblin', 4)]
```

第一批 4 只生成后 `_counters['goblin'] = 4`。第二批触发时，`already = _counters.get('goblin', 0) = 4`，
`count - already = 0` → **零只生成**。且条件 `already >= count` 满足，`_spawn_index` 推进，
`all_spawned` 变为 True。但只生成了 4 只，`enemies_killed` 最多 4，达不到 `total_enemies=8`，
波次永远卡住。

**为什么第 1、3 波不受影响？** 第 1 波只有一个 spawn 事件，不存在跨事件计数器污染。第 3 波
两批类型不同（goblin vs orc），即使 key 只用 etype 也不会冲突。只有**同类型多批次**的波次
（第 2、4、6、7、8、9、10 波）受影响。

**修复：** 将 key 从 `etype` 改为 `(spawn_index, etype)` 元组：
```python
key = (self._spawn_index, etype)
```
`(0, 'goblin')` 和 `(1, 'goblin')` 是不同的 key，两批独立计数。

**教训：** key 的粒度必须匹配业务语义。"看起来通用"的代码（复用 etype 做 key）在边界条件
翻车——它的正确性依赖隐含假设"同波次内每种敌人只出现一次"，第 2 波打破了这个假设。测试必
须覆盖"多 spawn 事件 + 同类型"的边界 case。

### 第 18 轮 — Bug 修复：敌人到基地不死 + 血条回满（2026-07-23）

**我:** 怪物被打但不死，血量又变满血。还有血条看不出减少。

**AI:** 两个关联问题。

**问题 1 — 基地漏怪不计入完成：**
`game.py` 的 `_update_enemies()` 中，敌人到达基地后调用 `enemy.kill()` 将 `_alive` 设为
False，但 `_hp` 未归零。在 kill 记录循环中 `if not enemy.alive and enemy.hp <= 0`——
到达基地的敌人 hp > 0，条件为假，**不计入 enemies_killed**。

后果链：`enemies_killed` 达不到 `total_enemies` → `Wave.completed` 永远 False → 波次卡死。

**修复：**
```python
# game.py _update_enemies()
elif enemy.alive and enemy.has_reached_base():
    self._base_hp -= enemy.damage_to_base
    enemy._hp = 0          # 关键：hp 归零
    enemy.kill()
    if self._current_wave:
        self._current_wave.enemies_killed += 1  # 计入完成统计
```

**问题 2 — 血条不可见/不变化：** 三原因：
1. 血条仅在 `hp_ratio < 1.0` 时绘制——满血无条，分不清"新敌人"和"满血老敌人"
2. 血条太细（0.06 单位高），且始终绿色——HP 从 100% 降到 70% 时视觉差异小
3. 新敌人在列表中靠后，画在最上层，盖住前线受伤敌人的缩水血条

**修复：**
1. 移除 `if hp_ratio < 1.0` 条件——血条始终显示
2. 加粗至 0.10 单位 + 颜色编码：绿(HP>60%) → 黄(30-60%) → 红(<30%)
3. 渲染前按 `path_progress` 升序排列——前线敌人（进度大）画在最上层

### 第 19 轮 — 战斗节奏修复：塔冷却错开（2026-07-23）

**我:** 算一下前几波数值。血条刚开始打不掉然后一下掉很多，很奇怪。

**AI:** 做了完整的时序模拟和根因分析。

**根因（5 层链条）：**

1. **固定时间步长 dt=0.08s + 塔初始冷却都是 0.0 → 第一帧就同步开火**
2. **冷却重置是"赋值"而非"累加"：** `self._cooldown_remaining = self.attack_cooldown` 把小数偏移信息丢失了，导致后续永久同步
3. **高弹速导致命中与发射可能同帧**（箭矢 12.0 单位/秒，dt=0.08s，0.96 格内立即命中）
4. **冷却帧边界效应：** 箭塔冷却 0.5s 需 7 帧（0.56s），两塔同步后每 7 帧同时开火，中间 6 帧全闲置
5. **视觉认知不协调：** HP 60→24（同帧）→ 0.56s 空窗 → 24→0（同帧）。阶梯式而非平滑。

**修复前时序：**
```
t=0.00s  双塔齐射
t=0.24s  双箭同时命中 → HP 60→24 (-60%！)
t=0.24~0.80s  空窗期（血条不动）
t=0.80s  双箭再次同时命中 → HP 24→0
```

**修复：** 在 Tower 构造函数中按网格位置设置初始冷却偏移：
```python
offset = ((col * 7 + row * 13) % 100) / 100.0
self._cooldown_remaining = offset * self.attack_cooldown
```

**修复后时序：**
```
塔A偏移=0.365s, 塔B偏移=0.04s
t=0.06s  塔B命中 → HP 60→42 (70% 绿)
t=0.35s  塔A命中 → HP 42→24 (40% 黄)
t=0.56s  塔B命中 → HP 24→6  (10% 红)
t=0.85s  塔A命中 → HP 6→0   (死)
```
每 0.2-0.3s 一次伤害，血条绿→黄→红平滑过渡。不再有 0.56s 空窗。

**我:** 手感明显改善。
